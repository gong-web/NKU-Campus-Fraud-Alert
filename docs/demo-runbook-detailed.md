## 1. 演示前准备

操作：

1. 在项目根目录执行 `docker compose up -d --remove-orphans`。
2. 执行 `docker compose exec -T backend alembic upgrade head`。
3. 执行 `docker compose exec -T backend python -m app.infra.db.seed`。
4. 如需展示统一入口，执行 `docker compose --profile edge up -d edge`，访问 `https://localhost:8443/healthz`。
5. 如需展示监控面板，执行 `docker compose --profile observability up -d prometheus grafana`。
6. 打开前端 `http://localhost:5173`。

现场策略：

- 主线只新建 1 条学生上报，用于展示“学生低门槛提交”和生成案件编号。
- 案件审核、匿名司法协助、测验统计优先使用 seed 预置数据，减少现场临时造数据的风险。
- 四个角色尽量分开浏览器或浏览器配置文件登录，避免同一个 Cookie 被不同角色覆盖。

讲解：

“这个项目是前后端分离的校园反诈平台。演示环境使用 Mock CAS，不需要接真实统一身份认证。后端、前端、MySQL、Redis 和 MinIO 基础服务都通过 Docker Compose 一键拉起。当前 Docker 演示环境中，证据文件会先加密，再写入 MinIO/S3 对象存储；测试环境也可以切换为本地加密目录。”

验收点：

- 前端地址可打开。
- 后端 `http://localhost:8000/healthz` 返回 `{"status":"ok"}`。
- 快捷登录页能看到学生、院系审核员、校级审核员、系统管理员四类账号。

### 1.1 架构图讲解的准确话术

建议按图从上到下这样介绍：

> 学生和管理人员通过浏览器访问 Vue 3 前端，前端负责交互、表单校验和状态
> 展示。普通演示可以通过 HTTP 直接访问 Vite 和 FastAPI；如需展示统一入口，
> 可启用 `edge` profile，由 Nginx 接收 HTTPS 请求，完成 TLS 终止和反向代理，
> 避免应用服务直接暴露。
>
> 请求随后进入 FastAPI 应用层。登录会话保存在共享 Redis 中，不依赖某一台
> 应用服务器，因此具备横向扩展的基础。生产环境配置多个应用实例、健康检查
> 和负载均衡后，单个应用实例异常时可由其他实例继续处理请求。当前 Docker
> Compose 演示实际运行的是单个 FastAPI 容器。
>
> 数据层当前使用单实例 MySQL 处理读写，Redis 负责在线会话、权限缓存和部分
> 异步审计。生产目标可以扩展为 MySQL 主库和只读副本，但本项目目前没有实现
> 主从复制与读写分离。
>
> 证据文件在写入存储前会进行 AES-256-GCM 加密。Docker 演示环境默认写入
> MinIO/S3，测试环境可切换到本地加密目录；这样既避免把大文件放进关系数据库，
> 也便于在没有对象存储的测试环境中快速验证功能。
>
> 密钥管理采用可切换的 KMS 抽象。演示环境使用 LocalKMS，从环境变量主密钥
> 派生数据密钥；Vault 和 AWS KMS Provider 当前是占位实现，正式部署必须接入
> 独立 KMS，使密钥与密文分离。系统已经实现 Real CAS 3.0 Provider，但本地
> 演示默认使用 Mock CAS，不能表述为已经接通学校真实 CAS。
>
> 可观测性方面，后端已提供 Prometheus `/metrics` 指标端点，并输出结构化
> 日志；Prometheus 和 Grafana 可通过 `observability` profile 启动，集中日志
> 平台仍属于生产部署接入项。
> 指标和应用日志采集可以旁路进行，但高敏审计会与业务事务同步写入，失败时
> 会阻止操作完成，因此不能把所有审计都描述成完全不阻塞主链路。

逐句核验：

| 原表述 | 结论 | 演示时如何修正 |
| --- | --- | --- |
| 浏览器访问 Vue 3，前端负责交互、校验和展示 | 准确 | 可原样保留 |
| HTTPS 进入 Nginx，完成 TLS 终止和反向代理 | 可选演示 / 生产目标 | 普通演示不启用；需要展示时启动 `edge` profile |
| 请求进入 FastAPI 集群 | 仅生产目标 | 当前是单容器；生产镜像支持 Gunicorn 多 worker，但没有部署多节点负载均衡 |
| 会话不固定在某台服务器 | 基础能力已实现 | 会话在 Redis，可不使用粘性会话；高可用仍依赖负载均衡和 Redis HA |
| MySQL 主写从读 | 尚未实现 | 当前单 MySQL、单连接地址；不要声称已有读写分离 |
| Redis 负责会话和缓存 | 准确但不完整 | 还承担 RBAC 缓存、CAS ticket 去重和审计 Stream |
| MinIO/S3 保存证据 | 准确 | Docker 演示默认 `STORAGE_BACKEND=s3`，证据加密后写入 MinIO；测试可切本地 |
| KMS 统一管理密钥 | 仅抽象与本地实现完成 | 当前 LocalKMS；Vault/AWS KMS 仍是占位 |
| CAS 对接校园统一认证 | 接口实现、真实接入未完成 | Real CAS Provider 已实现，演示使用 Mock CAS |
| Prometheus、Grafana和日志系统已部署 | 部分实现 | `/metrics` 已实现；Prometheus/Grafana 可用 profile 启动；集中日志需生产接入 |
| 虚线组件不阻塞主链路 | 不能概括审计 | 指标采集可旁路；高敏同步审计属于主事务的一部分 |

如果答辩老师追问“如何把生产架构真正落地”，按以下顺序回答：

1. 正式部署 Nginx 配置、域名和可信 TLS 证书，只对外暴露 443。
2. 部署至少两个 FastAPI 实例并配置健康检查；把定时任务拆为独立 worker 或增加分布式锁。
3. 配置 MySQL 复制和只读连接池，在仓储层区分主库事务与可最终一致的查询。
4. 验证 MinIO/S3 桶权限、生命周期、备份策略和跨环境迁移。
5. 实现 Vault/AWS KMS Provider，启用密钥轮换并禁止生产使用 LocalKMS。
6. 部署 Redis Sentinel/Cluster 和集中日志平台，完善 Prometheus/Grafana 告警规则。

## 3. 学生端：疑似诈骗上报

### 3.1 填写上报表单

操作：

1. 点击顶部“我要上报”。
2. 在“标题”输入：`演示上报-202606121727-冒充客服退款`。
3. “诈骗类型”选择“冒充客服退款类”。
4. “事发日期”选择当天或最近日期。
5. “涉案金额”输入 `1299.00`。
6. “诈骗手法简述”输入：`冒充平台客服，要求屏幕共享并索要验证码`。
7. “经过描述”输入完整案情。
8. 点击“点击上传图片”，选择一张证据图片。
9. 勾选“匿名上报”。

建议现场输入的经过描述：

“今天下午收到自称电商平台客服的电话，对方声称我的订单存在质量问题，可以办理三倍退款。随后对方要求添加聊天账号、打开屏幕共享，并让我输入银行卡号和短信验证码。我没有继续操作，保留了聊天截图和通话记录，怀疑这是冒充客服退款类诈骗。希望平台尽快核实并提醒同学不要开启屏幕共享。”

讲解：

“表单只要求标题、类型、日期和经过描述，金额、手法、联系方式和证据都可以补充。这里勾选匿名上报后，审核员不能直接看到学生身份，身份映射会被加密隔离。”

提交前务必确认：

- 标题至少填写 1 个字符，最多 200 字。
- 经过描述至少填写 1 个字符，最多 5000 字。
- 诈骗类型和事发日期必须选择。
- 涉案金额可以不填；填写时必须大于等于 0，最多保留两位小数。
- 证据不是必填项，不上传证据也可以提交。

导出故障排查：

- 正常结果是浏览器下载 CSV，页面不会跳走。
- 如果页面跳到 `localhost` 的 HTTP 500，检查前端容器日志是否包含 `http proxy error` 和 `ECONNREFUSED`。
- Docker 环境的 Vite 代理目标必须是 `http://backend:8000`，不能写容器内的 `http://localhost:8000`。
- CSV 使用 UTF-8 BOM，Excel 可直接识别中文；导出会沿用当前操作类型、对象类型、对象编号和操作人筛选条件。

其他导出与文件查看检查：

| 功能 | 正常结果 | 常见异常与处理 |
| --- | --- | --- |
| 测验完成率报告 XLSX | 下载 `.xlsx`，可打开“完成率报告”工作表 | 后端日志出现 `No module named openpyxl` 时，重新构建后端镜像；Dockerfile 已固定安装 `openpyxl` |
| 草稿证据查看 | 当前学生可打开本人草稿证据 | 404 通常表示草稿或文件编号不匹配；403 表示不是草稿所有者 |
| 审核证据查看 | 有证据查看权限的审核员确认敏感访问后打开文件 | 未带敏感访问确认头会被拒绝，这是权限保护，不是下载故障 |
| 中文下载文件名 | 浏览器使用响应中的中文或业务文件名 | 跨域环境需暴露 `Content-Disposition`；当前后端已配置 |

快速检查命令：

```powershell
docker compose exec -T backend python -c "import openpyxl; print(openpyxl.__version__)"
docker compose logs --tail=100 backend frontend
```

若 `docker compose up -d --build backend` 因 Docker Hub DNS 或临时网络问题无法重建，但后端容器仍在运行，可先执行下面的临时兜底命令保证现场导出功能可用；答辩后再在网络正常时重新构建镜像：

```powershell
docker compose exec -T backend uv pip install --python /opt/venv/bin/python "openpyxl>=3.1,<4.0"
```

## 11. 演示时容易被问到的问题

问：匿名上报是否真的匿名？

答：“普通审核员看不到真实身份。身份映射单独加密保存，只有系统管理员在司法协助流程中填写文书编号和理由，并在 5 分钟窗口内才能查看，且全过程写入审计。”

问：为什么有院系审核员和校级审核员？

答：“院系审核员负责日常案件处理和普通风险预警；校级审核员拥有更高权限，例如紧急预警、知识条目审核和全校范围数据。”

问：预警到期后会自动下线吗？

答：“当前页面支持填写失效时间，数据字段也保存了失效时间；但代码中暂未发现自动下线调度任务。现场演示建议用手动下线，并把自动下线作为后续增强点说明。”

问：测验和知识库有什么关系？

答：“题库题目可以关联知识库条目。学生答错后可以直接跳到对应知识条目学习，从测评变成复盘。”

问：审计是否可篡改？

答：“审计日志采用追加写和链式哈希校验，平台还提供 `make verify-audit` 用来校验审计链完整性。导出审计日志本身也会被审计。”

## 12. 演示故障排查与防错清单

### 12.1 “提交上报”无法点击或点击后未提交

典型现象：

![上报字段未达到最小长度](assets/demo-runbook/28-report-submit-disabled-invalid-fields.png)

旧版页面里，这类问题通常不是服务器故障，而是表单校验没有通过：

- 旧版标题和经过描述有较长字数限制。
- 当前演示版已降低字数门槛，标题和经过描述填写 `1` 也可以通过字数校验。
- 诈骗类型、事发日期和金额 `0.02` 都是有效的。
- 诈骗手法和联系方式是可选项，填写 `1` 不会阻止提交。
- 证据图片是可选项，是否上传不会影响提交按钮。

修复方法：

1. 确认标题不是空白。
2. 确认经过描述不是空白。
3. 确认诈骗类型已选择。
4. 确认事发日期已选择。
5. 再点击“提交上报”。

当前页面会提示：

- 标题为空时显示“请输入标题”。
- 描述为空时显示“请输入事件经过”。
- 按钮上方汇总还未完成的字段。
- 点击提交时会把焦点移到第一个不合格字段，不再静默无响应。

修复后的页面效果：

![上报表单实时校验与错误汇总](assets/demo-runbook/29-report-validation-feedback.png)

演示时可以这样说明：

“这里故意只输入一个字也能通过字数门槛。系统现在只拦截空白必填项、未选择类型或日期、金额格式错误等真正影响提交的数据；如果按钮有反应但没有产生案件，先看红色校验提示，不要重复点击。”

### 12.2 点击提交后提示 401 或跳回登录页

原因：

- 登录会话已过期。
- 在同一个浏览器配置文件中登录了另一个角色，原角色 Cookie 被覆盖。

处理：

1. 重新登录对应账号。
2. 多角色演示时使用不同浏览器、不同浏览器用户配置文件或独立无痕会话。
3. 不要在同一个 Chrome 普通窗口的不同标签页登录不同角色。

推荐窗口分配：

| 浏览器环境 | 账号 |
| --- | --- |
| Chrome 普通配置文件 | `student001` |
| Edge | `reviewer_dept001` |
| Chrome 无痕窗口 | `reviewer_school001` |
| Firefox 或第二个浏览器配置文件 | `sysadmin001` |

### 12.3 页面打开但数据一直加载失败

检查顺序：

1. 执行 `docker compose ps`。
2. 确认 backend、mysql、redis、minio 为 `healthy`。
3. 打开 `http://localhost:8000/healthz`，应返回 `{"status":"ok"}`。
4. 若容器未运行，执行 `docker compose up -d --remove-orphans`。
5. 若账号或字典数据缺失，执行：

```powershell
docker compose exec -T backend alembic upgrade head
docker compose exec -T backend python -m app.infra.db.seed
```

### 12.4 诈骗类型下拉框为空

可能原因：

- 后端没有运行。
- 数据库迁移或种子数据未执行。
- 当前登录会话失效。

处理：

1. 检查 `/healthz`。
2. 重新执行迁移和 seed。
3. 刷新页面并重新登录。

### 12.5 证据文件无法上传

规则：

- 单个案件最多 10 个证据文件。
- 单个文件不能超过 5 MB。
- 支持 JPEG、PNG、GIF、WebP、BMP、TIFF 和 PDF。

常见问题：

- 文件大于 5 MB。
- 浏览器或系统把文件识别成不支持的 MIME 类型。
- 已达到 10 个文件上限。
- 后端文件存储目录不可写。

演示建议：

- 提前准备一张 100 KB 至 1 MB 的 PNG 或 JPEG。
- 文件名使用简单中文或英文，不使用过长文件名。
- 正式答辩前先上传一次，确认预览和提交都正常。

#### 点击提交后出现 `Network Error`

如果证据已经上传，点击提交时却出现 `Network Error`，先不要连续重复点击。旧版后端在读取中文文件名的证据时，会把中文名称直接写入 HTTP 响应头，触发 Latin-1 编码异常并返回 500；浏览器有时只把它显示成笼统的 `Network Error`。当前版本已改为 RFC 5987 UTF-8 文件名编码，中文文件名可以正常预览和提交。

现场排查步骤：

1. 先打开“我的上报”，检查是否已经生成案件编号；提交请求可能已经成功，只是随后的证据预览请求失败。
2. 查看后端日志：

```powershell
docker compose logs backend --tail 100
```

3. 如果日志含有 `UnicodeEncodeError`、`latin-1` 和 `Content-Disposition`，说明运行的还是旧版代码。更新代码并重启后端：

```powershell
git pull
docker compose up -d --build backend
```

4. 来不及更新时，可临时把证据文件改成纯英文名，例如 `evidence-01.png`，删除旧证据后重新上传。
5. 如果“我的上报”里已经有案件编号，不要再次提交，避免产生重复案件。

本次故障的判断口诀可以这样介绍：

“看到 `Network Error` 不等于案件一定没提交。先查我的上报是否已有案件编号，再看后端日志；这样既能定位中文文件名兼容问题，也能避免重复上报。”

### 12.6 点击“保存草稿”后看不到草稿

处理：

1. 确认浏览器没有切换到其他学生账号。
2. 点击顶部“草稿箱”后手动刷新一次。
3. 检查是否弹出“草稿已保存”。
4. 检查后端是否正常。

说明：

“草稿属于当前学生，其他学生无法查看。草稿更新会刷新 30 天有效期。”

### 12.7 审核队列中找不到刚提交的案件

检查：

1. 确认学生提交成功页已经生成案件编号。
2. 审核队列状态筛选包含“待审核”和“审核中”。
3. 清空旧关键词，使用案件标题或案件编号搜索。
4. 确认当前院系审核员与学生属于同一院系。
5. 校级审核员可用于排除院系范围问题。

本演示中：

- `student001` 属于计算机学院。
- `reviewer_dept001` 也是计算机学院审核员。

### 12.8 案件进入详情后处理按钮不显示

处理动作只在 `REVIEWING` 状态显示。

可能原因：

- 案件已被其他审核员办结、驳回或转报警。
- 页面数据仍是旧状态。
- 当前账号没有 `report:review` 权限。

处理：

1. 刷新详情页。
2. 返回审核队列检查当前状态。
3. 换一条待审核案件重新演示。

#### 录入案例库弹窗点击“确认提交”看似无反应

典型现象：在“录入知识库案例”弹窗中，三个必填项都填了 `1`，点击“确认提交”后没有办结成功。

旧版原因：请求已经发到后端，但后端会按较长字数校验，返回 422 参数校验失败；页面又没有把原因显示清楚，所以看起来像“没反应”。

当前演示版已放开字数下限：

- 脱敏案例摘要、识别要点、防范建议、驳回原因和转报说明，填 `1` 也可以提交。
- 内部备注仍然可选。
- 若仍然出现 422，说明后端还停留在旧版本，执行 `git pull` 后重新构建后端。

为了让答辩更自然，正式演示仍建议填稍完整的示例：

- 脱敏案例摘要：`学生收到冒充电商客服的退款电话，对方诱导其开启屏幕共享并索要验证码。`
- 识别要点：`核实来电身份；拒绝屏幕共享；验证码不得告知他人。`
- 防范建议：`遇到退款理赔先通过官方渠道核验，切勿转账或泄露验证码。`

### 12.9 院系审核员无法发布紧急预警

这是正常的权限规则：

- 院系审核员可以发布提示级和警告级预警。
- 紧急级预警只能由校级审核员发布。

处理：

1. 院系审核员演示普通预警。
2. 切换 `reviewer_school001` 演示紧急预警。

### 12.10 知识条目没有“通过发布”按钮

检查：

- 条目必须处于 `PENDING` 待审核状态。
- 当前账号必须是校级审核员并拥有 `kb:review` 权限。

正确顺序：

1. `reviewer_dept001` 创建草稿并提交审核。
2. `reviewer_school001` 打开同一条目。
3. 点击“通过发布”或“驳回”。

### 12.11 学生看不到刚发布的知识条目

检查：

1. 管理端状态是否已经是“已发布”。
2. 学生端是否仍停留在旧列表缓存。
3. 清空搜索和诈骗类型筛选。
4. 刷新知识库列表。
5. 确认条目没有被下线。

### 12.12 随机测验无法开始

可能原因：

- 题库中启用题目不足 10 道。
- 当前账号不是学生。
- 登录会话过期。

处理：

1. 执行 seed，恢复示例题库。
2. 使用 `student001` 登录。
3. 确认管理员题库中至少有 10 道启用题。

### 12.13 指定测验无法作答

检查：

- 测验状态必须为 `ACTIVE`。
- 截止时间不能已经过去。
- 学生必须在全校、目标院系或指定学生范围内。
- 已提交的指定测验不能重复提交。

### 12.14 司法协助页面解密失败

可能原因：

- 输入的事件不是匿名上报。
- 事件没有对应匿名映射数据。
- 5 分钟窗口已经过期。
- 当前操作人不是原申请人。
- 当前账号不是系统管理员。

演示建议：

- 直接使用 seed 预置匿名事件 ID `990000000000000005`，关联案件编号为 `2026-CS-900005`。
- 先讲清申请理由和文书编号，再提交。
- 提交后立即执行窗口内解密，不要中途切换账号。

### 12.15 截图或页面样式没有更新

处理：

1. 浏览器执行强制刷新：`Ctrl + F5`。
2. 确认打开的是 `http://localhost:5173`。
3. 查看 `docker compose logs --tail=100 frontend`。
4. 必要时执行：

```powershell
docker compose restart frontend
```
