# ADR-003: 不引入 GraphQL

## 背景

GraphQL 在前端取数据上有"按需查询、聚合多资源"的优势。本项目要不要上？

## 决策

**不引入**。坚守 RESTful + OpenAPI + 自动生成前端类型。

## 后果

**好处**

- 路径风格统一（PRD 4.6.2），一目了然。
- 鉴权点清晰：每个 controller 一个 `Depends(require_*)`，GraphQL 的 field-level resolver 鉴权远复杂。
- 学校防火墙 / 审计设备对 REST 路径更友好，便于运维定位。
- 前端 axios + openapi-typescript-codegen 已能解决"字段名拼错"。

**代价**

- 个别复杂页面（如审核员工作台）可能多调几个接口；可通过 BFF 接口聚合解决。

## 替代方案

| 方案       | 否决原因                                                |
| ---------- | ------------------------------------------------------- |
| GraphQL    | 鉴权 / 审计 / 限流复杂度激增；课程一年内实现风险高。     |
| tRPC       | 强绑 TypeScript 客户端；与 PRD 多端访问目标不符。        |
