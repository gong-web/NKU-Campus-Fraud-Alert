配套精细化演示手册：[`demo-runbook-detailed.md`](demo-runbook-detailed.md)。该手册包含实际跑通截图、逐步点击说明和现场讲解话术。

## 2. 演示前准备

- [ ] 优先使用预置演示数据跑主线；现场只新建 1 条学生上报即可，避免临时造太多数据导致查找困难。
- [ ] 执行 `make up` 启动 MySQL、Redis、MinIO、后端和前端。
- [ ] 执行 `make migrate && make seed` 完成迁移和种子数据初始化。
- [ ] 确认前端 `http://localhost:5173` 可访问。
- [ ] 确认后端 `http://localhost:8000/healthz` 返回 `{"status":"ok"}`。
- [ ] 如需展示统一入口，执行 `docker compose --profile edge up -d edge`，访问 `https://localhost:8443/healthz`。
- [ ] 如需展示监控面板，执行 `docker compose --profile observability up -d prometheus grafana`，访问 Prometheus `http://localhost:9090` 和 Grafana `http://localhost:3000`。
- [ ] 确认开发快捷登录页能看到四种角色。
- [ ] 准备一张小于 5 MB 的图片，用于证据上传演示。
- [ ] 确认学生端可看到 `2026-CS-900001` 等预置演示案件。
- [ ] 司法协助演示使用事件 ID `990000000000000005`，无需手动写匿名映射。
- [ ] 多角色演示时使用不同浏览器或不同用户配置文件，避免 Cookie 覆盖导致突然跳回登录页。

种子数据已包含账号与权限、案件全状态样本、草稿、不同等级预警、知识审核状态、反诈题库、测验历史、错题和站内通知。重复执行 seed 不会重复插入；如需恢复被操作过的初始状态，应在确认可清库后执行 `make reset`。

### 2.1 架构讲解边界

- [ ] 明确普通演示使用 HTTP、Vite 和单个 FastAPI 容器；如需展示统一入口，可用 `edge` profile 启动 Nginx/TLS。
- [ ] 明确当前 MySQL 和 Redis 都是单实例，没有实现主从读写分离或 Sentinel。
- [ ] 明确 Docker 演示环境证据会先加密再写入 MinIO/S3；测试环境可切换为本地目录。
- [ ] 明确当前使用 LocalKMS 和 Mock CAS；Vault/AWS KMS、学校真实 CAS 属于生产接入项。
- [ ] 明确已提供 Prometheus `/metrics` 和结构化日志；Prometheus/Grafana 可用 `observability` profile 启动，ELK/Loki 属于生产接入项。
- [ ] 不把高敏同步审计描述为旁路；它失败时会阻止高敏业务操作完成。
- [ ] 介绍应用多节点、MySQL 主从、Redis HA、外部 KMS 和集中日志时使用“生产目标架构”措辞。

## 3. 推荐演示主线

建议采用“学生发现风险 → 上报 → 管理员审核 → 发布预警/沉淀知识 → 学生学习测验 → 系统审计追溯”的业务闭环。

稳妥版建议只现场新建 1 条学生上报，其余闭环优先使用 seed 预置数据：

1. 学生登录，浏览首页预警与知识推荐。
2. 学生填写匿名上报、上传证据并提交，记录现场生成的案件编号。
3. 院系审核员处理预置案件 `2026-CS-900003`，避免刚提交案件因筛选条件或状态刷新影响节奏。
4. 校级审核员发布紧急预警，学生收到通知。
5. 院系审核员创建知识条目，校级审核员审核发布。
6. 学生搜索知识并完成一次随机练习或查看预置测验结果。
7. 系统管理员查看账号、审计记录、CSV 导出和司法协助流程。

## 10. 25 分钟现场演示脚本

### 第一段：学生上报，约 5 分钟

1. 使用 `student001` 登录。
2. 展示首页最高级预警、知识推荐和反诈热线。
3. 进入“我要上报”，填写一条匿名诈骗事件。
4. 上传证据，先保存草稿。
5. 从草稿箱继续编辑并提交。
6. 记录生成的案件编号。

### 第二段：案件审核，约 5 分钟

1. 切换 `reviewer_dept001`。
2. 在审核队列按案件编号搜索。
3. 打开详情，说明案件由待审核自动进入审核中。
4. 二次确认后查看证据。
5. 展示匿名上报者信息隔离。
6. 选择“录入案例库并办结”或“驳回”，完成状态流转。

### 第三段：预警和知识闭环，约 5 分钟

1. 使用院系审核员发布一条院系定向普通预警。
2. 使用院系审核员创建知识草稿并提交审核。
3. 切换 `reviewer_school001`，通过知识审核。
4. 再发布一条全校紧急预警，说明紧急级权限限制。
5. 切回学生，查看通知、预警和新发布知识条目。

### 第四段：反诈测验，约 5 分钟

1. 学生开始随机练习或示例指定测验。
2. 故意答错若干题后提交。
3. 展示成绩和逐题解析。
4. 从错题跳转知识库。
5. 展示错题本和测验历史。
6. 切换审核员查看完成率并导出 Excel。

### 第五段：系统治理，约 5 分钟

1. 使用 `sysadmin001` 登录。
2. 展示系统概览和账号列表。
3. 新建一个测试账号。
4. 查询刚才产生的登录、上报、审核、预警等审计记录。
5. 导出审计 CSV。
6. 点击“填入演示数据”，展示司法协助申请、约 `05:00` 倒计时、身份结果、水印和审计记录。
